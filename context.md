# Project Context (Alcora Clinic Appointments Dashboard)

## Purpose
This repo contains a Vite + React + TypeScript web app for Alcora Labs. It has:
- A public landing page (marketing site).
- A clinic dashboard for reception teams to send WhatsApp reminders, manage follow-ups, and track daily completion.
- A placeholder auth page (routes to the dashboard).

The dashboard is a front-end shell that pulls data from Supabase Edge Functions. Those functions are expected to read/write each clinic's Google Sheets.

## Core Workflow (Reception)
- Outbox messages are generated by automation (n8n) and stored in Sheets.
- Receptionists click "Send WhatsApp" to open a prebuilt `waLink` and then mark it as done.
- Follow-ups are handled in separate queues (Cancelled, Reschedule, Recall).

## Data Sources
- Supabase project: `pseelatiesjoilszzaej`

  - `get_patients?tab=<SheetTab>` (fetches a sheet tab)

## Context: Dental Clinic WhatsApp Appointment System (UAE)

### 0) What this is
A practical appointment confirmation + reminder + follow-up system for UAE dental clinics.

Phase 1 (current): **Assisted automation**
- n8n generates messages and links.
- Receptionist clicks a **WhatsApp Web button** to open the chat with pre-filled text.
- Receptionist presses send in WhatsApp.
- The click is logged so the dashboard stays accurate.

Phase 2 (later): **Fully automated**
- WhatsApp Business API sends automatically.
- Dashboard becomes mostly view-only + exception handling.

### 1) Reality: what the receptionist does day-to-day
**Booking happens**
1) Patient calls clinic.
2) Receptionist books appointment in **Google Calendar**.
3) Receptionist includes patient details in the event description (free text):
   - patient name
   - phone number
   - dentist/doctor
   - any notes

**Sending happens (Phase 1)**
4) Dashboard shows a **Send WhatsApp** button for that appointment message.
5) Receptionist clicks it; WhatsApp opens with the message prefilled.
6) Receptionist presses send.
7) Dashboard logs status so nothing gets missed or duplicated.

**Follow-up happens**
8) If patient clicks Confirm / Cancel / Reschedule in the message, it gets routed to the right queue automatically.
9) Receptionist only handles exceptions:
   - cancellations call-back
   - reschedule requests call-back
   - recall / no-next visits (fail-safe list)

### 2) System components
**A) n8n (backend brain)**
- Scheduling (48h, 24h, 2h reminders)
- Calendar pulls
- Parsing/normalizing event description
- Message generation + WhatsApp link generation
- Idempotency/dedupe
- Routing patient actions from Tally webhooks
- Writing to storage (Sheets now; Supabase later)

**B) Storage (current: Google Sheets; later: Supabase)**
- Outbox messages
- Confirmations
- Follow-up queues
- Recall / no-next appointment list

**C) Front end dashboard (MVP)**
- Display send queue
- Display follow-up queues
- Provide buttons (WhatsApp, mark handled, add notes)
- Show simple metrics + weekly summary
- Open drawers (Appointment view, Patient view)

### 3) Data model (tables / sheets)
**3.1 Outbox (send queue)**
Each row = one message that should be sent.

Core columns:
- createdAt
- appointmentId
- messageType: `confirm | reminder_48hr | reminder_tomorrow | reminder_2h | book_next_nudge1 | book_next_nudge_manual`
- potentialDuplicate: TRUE/FALSE
- idempotencyKey (unique per message)
- phoneE164
- messageText
- waClickable (label)
- waLink (WhatsApp URL)
- sendStatus: `ready | opened | sent | needs_review`

Recommended operational columns (for clean UI + reliability):
- patient_name
- dentist
- startIso
- displayTime
- eventUpdatedAt
- openedAt, openedBy
- sentAt, sentBy
- notSentReason

Primary join key: `appointmentId`

**3.2 Confirmed**
Logged when patient confirms via link.
- createdAt, appointmentId, patient_name, phone, dentist, start, status

**3.3 CancelledFollowUp**
Logged when patient cancels via link.
- createdAt, appointmentId, patient_name, phone, dentist, start, cancelReason, aiSummary

Required queue fields:
- followupStatus: `open | contacted | booked | closed`
- handledBy, handledNote, updatedAt

**3.4 RescheduleFollowUp**
Logged when patient requests reschedule via link.
- createdAt, appointmentId, patient_name, phone, dentist, currentStart, note, aiSummary

Also requires:
- followupStatus, handledBy, handledNote, updatedAt

**3.5 Recall (no-next appointment list)**
Fail-safe list for patients who leave without booking next visit.
- createdAt, patient_name, phoneE164, dentist, lastVisitIso, note/service, status, idempotencyKey

### 4) n8n backend: workflow set (how it works)
**WF1: Booking confirmation (immediate)**
- Trigger: Google Calendar event created
- Parse details from event description, normalize phone -> E164
- Generate WhatsApp message + waLink
- Compute idempotencyKey: `{appointmentId}:{messageType}:{startIso}`
- Write row into Outbox (sendStatus=ready)
- If duplicate detected, set potentialDuplicate=true or sendStatus=needs_review

**WF2: 48-hour reminder**
- Trigger: daily 9:00 AM Dubai
- Pull events for ?2 days from now?
- Generate message + waLink
- Dedupe using idempotencyKey
- Write to Outbox

**WF3: 24-hour reminder (with actions)**
- Trigger: daily 9:00 AM Dubai
- Pull events for ?tomorrow?
- Generate message with 3 links (Confirm / Cancel / Reschedule)
- Links are Tally forms with hidden fields
- Write to Outbox with idempotency

**WF4: Patient action handling**
- Trigger: webhook from Tally submission
- Route by action:
  - Confirm -> write to Confirmed
  - Cancel -> write/upsert to CancelledFollowUp
  - Reschedule -> write/upsert to RescheduleFollowUp
- Last submission wins (upsert by appointmentId)

**WF5: 2-hour reminder**
- Trigger: every 15 minutes
- Pull events starting within next 2 hours
- Generate short reminder
- Write to Outbox with idempotency

**WF6: Recall capture (manual)**
- Trigger: receptionist marks a patient as ?no next appointment?
- Write to Recall list

### 5) How n8n is hooked to the front end
**Option A (current in this repo)**: Front end talks to Supabase Edge Functions
- The UI calls `get_patients?tab=<SheetTab>`
- Edge functions read/write Sheets and return JSON

**Option B (later)**: Front end talks to Supabase directly
- n8n writes to Supabase tables
- UI reads/writes Supabase for faster filtering

### 6) Front end logic (what the UI does)
**Send now**: Outbox rows with `sendStatus=ready` and no duplicates.
**Opened**: Outbox rows with `sendStatus=opened`.
**Needs review**: Outbox rows flagged duplicate or missing data.
**Follow-ups**: Cancelled + Reschedule + Recall queues.

### 7) WhatsApp button logging (critical)
Because WhatsApp Web has no delivery/sent API:
1) Open waLink in a new tab
2) Update Outbox: sendStatus=opened, openedAt, openedBy
3) Show modal: ?Did you send the message??
   - Yes: sendStatus=sent, sentAt, sentBy
   - Not sent: revert to ready or keep opened, capture notSentReason

### 8) Reliability rules
- Every message uses deterministic `idempotencyKey`
- Duplicates go to Needs Review
- ?Last submission wins? for patient actions (upsert by appointmentId)

### 9) Edge cases
- Appointment edited after message created -> show warning
- Missing/invalid phone -> block send + needs_review
- WhatsApp replies instead of links -> manual override
- Multiple staff clicks -> show openedBy/sentBy and lock row

### 10) Success criteria
Receptionist:
- Clears Send Now queue quickly
Owner:
- Checks weekly summary for rates and pending follow-ups
System:
- Reminders are never forgotten because the queue forces visibility
