# Project Context (Alcora Clinic Appointments Dashboard)

## Purpose
This repo contains a Vite + React + TypeScript web app for Alcora Labs. It has:
- A public landing page (marketing site).
- A clinic dashboard for reception teams to send WhatsApp reminders, manage follow-ups, and track daily completion.
- A placeholder auth page (routes to the dashboard).

The dashboard is a front-end shell that pulls data from Supabase Edge Functions. Those functions are expected to read/write each clinic's Google Sheets.

## Core Workflow (Reception)
- Outbox messages are generated by automation (n8n) and stored in Sheets.
- Receptionists click "Send WhatsApp" to open a prebuilt `waLink` and then mark it as done.
- Follow-ups are handled in separate queues (Cancelled, Reschedule, Recall).

  - `get_patients?tab=<SheetTab>` (fetches a sheet tab)

## Context: Dental Clinic WhatsApp Appointment System (UAE)

### 0) What this is
A practical appointment confirmation + reminder + follow-up system for UAE dental clinics.

Phase 1 (current): **Assisted automation**
- n8n generates messages and links.
- Receptionist clicks a **WhatsApp Web button** to open the chat with pre-filled text.
- Receptionist presses send in WhatsApp.
- The click is logged so the dashboard stays accurate.

Phase 2 (later): **Fully automated**
- WhatsApp Business API sends automatically.
- Dashboard becomes mostly view-only + exception handling.

### 1) Reality: what the receptionist does day-to-day
**Booking happens**
1) Patient calls clinic.
2) Receptionist books appointment in **Google Calendar**.
3) Receptionist includes patient details in the event description (free text):
   - patient name
   - phone number
   - dentist/doctor
   - any notes

**Sending happens (Phase 1)**
4) Dashboard shows a **Send WhatsApp** button for that appointment message.
5) Receptionist clicks it; WhatsApp opens with the message prefilled.
6) Receptionist presses send.
7) Dashboard logs status so nothing gets missed or duplicated.

**Follow-up happens**
8) If patient clicks Confirm / Cancel / Reschedule in the message, it gets routed to the right queue automatically.
9) Receptionist only handles exceptions:
   - cancellations call-back
   - reschedule requests call-back
   - recall / no-next visits (fail-safe list)

### 2) System components
**A) n8n (backend brain)**
- Scheduling (48h, 24h, 2h reminders)
- Calendar pulls
- Parsing/normalizing event description
- Message generation + WhatsApp link generation
- Idempotency/dedupe
- Routing patient actions from Tally webhooks
- Writing to storage (Sheets now; Supabase later)

**B) Storage (current: Google Sheets; later: Supabase)**
- Outbox messages
- Confirmations
- Follow-up queues
- Recall / no-next appointment list

**C) Front end dashboard (MVP)**
- Display send queue
- Display follow-up queues
- Provide buttons (WhatsApp, mark handled, add notes)
- Show simple metrics + weekly summary
- Open drawers (Appointment view, Patient view)

### 3) Data model (tables / sheets)
**3.1 Outbox (send queue)**
Each row = one message that should be sent.

Core columns:
- createdAt
- appointmentId
- messageType: `confirm | reminder_48hr | reminder_tomorrow | reminder_2h | book_next_nudge1 | book_next_nudge_manual`
- potentialDuplicate: TRUE/FALSE
- idempotencyKey (unique per message)
- phoneE164
- messageText
- waClickable (label)
- waLink (WhatsApp URL)
- sendStatus: `ready | opened | sent | needs_review`

Recommended operational columns (for clean UI + reliability):
- patient_name
- dentist
- startIso
- displayTime
- eventUpdatedAt
- openedAt, openedBy
- sentAt, sentBy
- notSentReason

Primary join key: `appointmentId`

**3.2 Confirmed**
Logged when patient confirms via link.
- createdAt, appointmentId, patient_name, phone, dentist, start, status

**3.3 CancelledFollowUp**
Logged when patient cancels via link.
- createdAt, appointmentId, patient_name, phone, dentist, start, cancelReason, aiSummary

Required queue fields:
- followupStatus: `open | contacted | booked | closed`
- handledBy, handledNote, updatedAt

**3.4 RescheduleFollowUp**
Logged when patient requests reschedule via link.
- createdAt, appointmentId, patient_name, phone, dentist, currentStart, note, aiSummary

Also requires:
- followupStatus, handledBy, handledNote, updatedAt

**3.5 Recall (no-next appointment list)**
Fail-safe list for patients who leave without booking next visit.
- createdAt, patient_name, phoneE164, dentist, lastVisitIso, note/service, status, idempotencyKey

### 4) n8n backend: workflow set (how it works)
**WF1: Booking confirmation (immediate)**
- Trigger: Google Calendar event created
- Parse details from event description, normalize phone -> E164
- Generate WhatsApp message + waLink
- Compute idempotencyKey: `{appointmentId}:{messageType}:{startIso}`
- Write row into Outbox (sendStatus=ready)
- If duplicate detected, set potentialDuplicate=true or sendStatus=needs_review

**WF2: 48-hour reminder**
- Trigger: daily 9:00 AM Dubai
- Pull events for ?2 days from now?
- Generate message + waLink
- Dedupe using idempotencyKey
- Write to Outbox

**WF3: 24-hour reminder (with actions)**
- Trigger: daily 9:00 AM Dubai
- Pull events for ?tomorrow?
- Generate message with 3 links (Confirm / Cancel / Reschedule)
- Links are Tally forms with hidden fields
- Write to Outbox with idempotency

**WF4: Patient action handling**
- Trigger: webhook from Tally submission
- Route by action:
  - Confirm -> write to Confirmed
  - Cancel -> write/upsert to CancelledFollowUp
  - Reschedule -> write/upsert to RescheduleFollowUp
- Last submission wins (upsert by appointmentId)

**WF5: 2-hour reminder**
- Trigger: every 15 minutes
- Pull events starting within next 2 hours
- Generate short reminder
- Write to Outbox with idempotency

**WF6: Recall capture (manual)**
- Trigger: receptionist marks a patient as ?no next appointment?
- Write to Recall list

### 5) How n8n is hooked to the front end
**Option A (current in this repo)**: Front end talks to Supabase Edge Functions
- The UI calls `get_patients?tab=<SheetTab>`
- Edge functions read/write Sheets and return JSON

**Option B (later)**: Front end talks to Supabase directly
- n8n writes to Supabase tables
- UI reads/writes Supabase for faster filtering

### 6) Front end logic (what the UI does)
**Send now**: Outbox rows with `sendStatus=ready` and no duplicates.
**Opened**: Outbox rows with `sendStatus=opened`.
**Needs review**: Outbox rows flagged duplicate or missing data.
**Follow-ups**: Cancelled + Reschedule + Recall queues.

### 7) WhatsApp button logging (critical)
Because WhatsApp Web has no delivery/sent API:
1) Open waLink in a new tab
2) Update Outbox: sendStatus=opened, openedAt, openedBy
3) Show modal: ?Did you send the message??
   - Yes: sendStatus=sent, sentAt, sentBy
   - Not sent: revert to ready or keep opened, capture notSentReason

### 8) Reliability rules
- Every message uses deterministic `idempotencyKey`
- Duplicates go to Needs Review
- ?Last submission wins? for patient actions (upsert by appointmentId)

### 9) Edge cases
- Appointment edited after message created -> show warning
- Missing/invalid phone -> block send + needs_review
- WhatsApp replies instead of links -> manual override
- Multiple staff clicks -> show openedBy/sentBy and lock row

### 10) Success criteria
Receptionist:
- Clears Send Now queue quickly
Owner:
- Checks weekly summary for rates and pending follow-ups
System:
- Reminders are never forgotten because the queue forces visibility

## Metrics Implementation Details

### Calculated KPIs

#### Reminder Coverage (by type)
- **Formula:** `count(Outbox where messageType = "reminder_48hr|reminder_tomorrow|reminder_2h")`
- **Why:** Ensures all scheduled reminders are being generated and sent
- **Calculation:** Simple count of outbox rows filtered by messageType
- **Status:** ✅ Fully implemented

#### Confirmation Rate (24h)
- **Formula:** `confirmed(24h) / sent(24h) * 100`
- **Why:** Key metric for patient engagement and message effectiveness
- **Calculation:** Count Confirmed rows created in last 24h, divide by Outbox rows with sentAt in last 24h
- **Status:** ✅ Fully implemented

#### No Response Rate (24h)
- **Formula:** `(sent - confirmed - cancelled - rescheduled) / sent * 100`
- **Why:** Identifies patients who received message but took no action
- **Calculation:** Subtract all response types from sent count
- **Status:** ✅ Fully implemented

#### Cancellation & Reschedule Rates
- **Formula:** `cancelledFollowUps.length / totalOutboxSent * 100`
- **Why:** Track appointment loss and identify issues
- **Calculation:** Count of all CancelledFollowUp/RescheduleFollowUp rows vs total Outbox sent
- **Status:** ✅ Fully implemented

#### Follow-up Completion Rate
- **Formula:** `(closedCancelled + closedReschedule) / totalFollowups * 100`
- **Why:** Measure speed of resolving patient responses
- **Calculation:** Count rows with followupStatus="closed" vs all rows
- **Status:** ✅ Fully implemented

#### Follow-up Response Time (Median)
- **Formula:** `median(updatedAt - createdAt) for closed follow-ups`
- **Why:** Benchmark team responsiveness
- **Calculation:** Extract time deltas from all closed follow-ups, sort, find median in hours
- **Status:** ✅ Fully implemented

#### Recovery Rate
- **Formula:** `cancelledWithRebook / totalCancelled * 100`
- **Why:** Measure if we successfully resell cancelled appointments
- **Calculation:** Count CancelledFollowUp where followupStatus="booked" divide by total cancelled
- **Status:** ✅ Fully implemented (depends on followupStatus="booked" value being used)

#### Open Follow-ups Backlog
- **Formula:** `count(where followupStatus="open" AND createdAt in range)`
- **Overdue:** `count(where age > 2 days)`
- **Why:** Revenue leakage indicator - unresolved follow-ups = lost rebooking
- **Calculation:** Filter open cancelled + reschedule + recall rows
- **Status:** ✅ Fully implemented

#### Recall Conversion Rate
- **Formula:** `recalled / ready * 100`
- **Why:** Measure effectiveness of "no-next-visit" follow-up system
- **Calculation:** Count Recall rows with sendStatus="recalled" vs "ready"
- **Status:** ✅ Fully implemented

#### Needs Review Rate (Data Quality)
- **Formula:** `needsReviewCount / totalOutbox * 100`
- **Why:** Identify upstream data entry issues
- **Calculation:** Count rows flagged for review (missing name/dentist/phone, duplicates) vs total
- **Status:** ✅ Fully implemented

#### Cancellation Rate Trend (MoM)
- **Formula:** `((current_month - previous_month) / previous_month) * 100`
- **Why:** Detect rising cancellation issues
- **Calculation:** Filter CancelledFollowUp by createdAt range, compute % change
- **Status:** ✅ Fully implemented

#### Time-Slot Leakage Analysis
- **Formula:** `GROUP BY startIso(hour) → (confirmed + cancelled + noResponse) / total`
- **Why:** Identify time slots with highest dropout rates
- **Calculation:** Group confirmations and cancellations by hour, compute leakage rate per hour
- **Status:** ✅ Fully implemented

#### Day-of-Week Patterns
- **Formula:** `GROUP BY day(startIso) → confirmation%, cancellation%, volume`
- **Why:** Identify weekly trends for scheduling optimization
- **Calculation:** Group by day of week, compute rates and counts
- **Status:** ✅ Fully implemented

#### Top Cancellation Reasons
- **Formula:** `frequency(CancelledFollowUp.cancelReason) sorted DESC, top 10`
- **Why:** Identify systemic issues (cost, availability, etc.)
- **Calculation:** Extract cancelReason field, count frequency, sort
- **Status:** ✅ Fully implemented

## What Cannot Be Computed (and Why)

### Message Type ROI
- **Issue:** Impossible to attribute a patient response to a specific reminder without explicit tracking
- **Example:** Patient receives 48hr + 24hr + 2hr reminders → confirms. Which one caused it?
- **Solution:** Would require each reminder to have a unique tracking ID and patient action to reference it
- **Status:** ⚠️ Not implementable without backend changes

### Dentist Revenue Impact
- **Issue:** Dashboard data doesn't include appointment fees or which appointments are revenue-generating
- **Reason:** Scope is WhatsApp messaging, not accounting
- **Status:** ⚠️ Out of scope (would require integration with accounting system)

### Message Response Time Distribution
- **Issue:** We only know when message was sent (sentAt) and when form was submitted (createdAt in followup sheet), not when message was read
- **Reason:** WhatsApp Web doesn't expose read receipts to third-party apps
- **Status:** ⚠️ Not feasible with current architecture

### Patient Response Patterns by Reminder Type
- **Issue:** Same as Message Type ROI - can't isolate which reminder triggered the response
- **Status:** ⚠️ Not implementable without unique tracking IDs per reminder

## How Metrics Are Calculated (Technical Details)

### Data Flow

1. **Data Fetch** (SkeletonDashboard.tsx)
   - useEffect hook polls Supabase Edge Functions every 30 seconds
   - Fetches: Outbox, Confirmed, CancelledFollowUp, RescheduleFollowUp, Recall sheets
   - Data is mapped to typed objects (OutboxRow, CancelledFollowUp, etc.)

2. **State Update**
   - Raw arrays stored in React state: `outboxRows`, `confirmRows`, `cancelledFollowUps`, `rescheduleFollowUps`, `recallRows`
   - Metrics state: `metricsData` (can be null while loading)

3. **Metrics Calculation**
   - Triggered by useEffect whenever any data array changes
   - Calls `calculateMetrics()` from metricsCalculator.ts
   - Pure function - no side effects
   - Returns `MetricsData` object with all KPIs

4. **Display**
   - Reports component receives `metricsData` as prop
   - Renders tabs and charts based on available metrics
   - Charts use Recharts library with responsive containers

### Date/Time Handling

- All dates assumed to be ISO 8601 strings in Sheets
- Parsed using `parseISO()` from date-fns
- Dubai timezone used for "today" checks via `toZonedTime()` from date-fns-tz
- Comparisons use UTC milliseconds for consistency

### Example Calculation: Confirmation Rate (24h)

```typescript
const now = new Date();
const last24h = new Date(now.getTime() - 24 * 60 * 60 * 1000);

const sentLast24h = outboxRows.filter((r) => 
  r.sentAt && isDateInRange(r.sentAt, last24h, now)
).length;

const confirmedLast24h = confirmRows.filter((r) => 
  r.createdAt && isDateInRange(r.createdAt, last24h, now)
).length;

const confirmationRate24h = sentLast24h > 0 ? (confirmedLast24h / sentLast24h) * 100 : 0;
```
